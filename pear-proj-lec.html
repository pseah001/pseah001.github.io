Background

  • Visit 11 different systems and documents

  • Takes ~ 1-3 hours

    Discovery & Define

    1. Unpacking & understanding the basic knowledge and workflow

    Situation/constraints

    As the project is very complicated, we find ourselves spending the bulk of discovery unpacking the basic knowledge and workflow.​​​​​​​

    Tasks

    Create a summarised documentation to collate our understanding which includes:​​​​​

  • Glossary page collating meaning of abbreviations used
  • Understanding JTBD for each users
  • Mapping of complex concepts: DOA, facilities & collaterals
  • High level & flow

    • Results

    We find ourselves referring to the documentation very often as its challenging to remember so much information given the complexity of the project.

    Newly hired designers also feedback that the summary eased their entry to the project.

    2. Aligning on the short & long term direction

    Situation 1

    Currently, separate systems are used to process different types of loan: cp (SFW) & lp (osca) & crs (glow). To find out what loan type an application is, user has to create it in one of the system. If a loan type is lp but created in SFW (used for cp loans), users have to redo the application in Lp’s system, osca. Thus, the goal was to have a single system to process different loan types.​​​​​​​

    Task 1: Make sure the end goal is correct

    To confirm if having 1 system for different loan type is the correct direction & if its technically feasible​​​​​

  • Discover the loan creation flow for each loan type, high level & detailed level

    • Results

  • Found out that process for cp & lp is similar. Both go through pre check to find out if a loan is cp or lp. Thus there is a possibility that flow may be combined in future.
  • Loan makers can easily identify that a loan application is a crs type at the start. Thus, there is no need for a combined flow.

    • Situation 2

    Business were not sure what the interim solution will be like & how to get there.​​​​​​​

    Task 2: Align on short term & long term direction

    Suggested possible short & long-term direction taking into consideration of the timeline.

    Results

    New direction:
  • Keep CRS separate but login for 3 loans will be the same. Due to time limitation, we decided to focus on LP journey first but keeping in mind that the flow may be combined in future.
  • MVP 1: focus on eligibility check tool for LP. ELigibility check for LP → case creation → submit case for approval
  • MVP 2: complete journey for LP case. Eligibility check tool for LP → case creation → submit case for approval → post approval flow
  • Future iterations: discover & add CP flow, discover & add CRS flow

    • Design

    1. From 11 systems to 4 steps

    Problem

  • Visit 11 different systems and documents
  • Manual calculation of individual information to view as group level
  • Takes around 1-3h
  • Case may not be converted at the end

    • Solution

    RMs are collecting and validating information here and there, and it may seem there's no sequence. But throughout the discovery, we realized that there's a set of information to see certain results for their customer's eligibility check.

    We categorise users' main tasks into:

  • Checking multiple sets of information to see if there is any issues.
  • Checking individual's information to clarify any uncertain points.
  • Checking each borrower's existing facilities to calculate the entire group limit.

    • We transformed the current flow into a sequential step so that users can identify necessary information to see all the desired results. Results with the same disposition were grouped and displayed together.

    Result

    4 steps journey with prevalidation results in 5 mins to determine if a case is worth creating.

    2. Auto-filling of data

    Problem

  • RM has to login to 11 systems, copy data each system and paste data into the case creation portal.
  • Manual input of up to 915 fields.
  • RM constantly worried that they fill in data inaccurately.

    • "It is so tedious, I have to toggle back and forth systems to make sure my input is accurate."

    Solution

    When case is created, data will be pulled back-end and auto-fill the case.

    Result

  • Eliminate the need for manual inputs.
  • RM will not be worried that they filled in the wrong data.

    • Biggest takeaway

    Importance of documentation

    It was easy to be information overloaded and feel overwhelmed given the scale of this project. Summaries and documentations were exceptionally important for this project as it provided a consolidated place for us to refresh our memory and focus on the task on hand.
